預設排程演算法：CFS (完全公平排程器)

Linux 核心的預設 CPU 排程演算法是 CFS (Completely Fair Scheduler，完全公平排程器),用於 SCHED_NORMAL (也稱為 SCHED_OTHER) 排程策略。 sched-design-CFS.rst:18-19

核心概念

CFS 的設計理念是在真實硬件上模擬一個「理想的、精確的多任務 CPU」,讓每個任務以相同速度並行運行。 sched-design-CFS.rst:21-29 它使用虛擬運行時間 (vruntime) 來追蹤每個任務的執行時間,並總是選擇 vruntime 值最小的任務運行。 sched-design-CFS.rst:36-44

實作細節

CFS 使用紅黑樹資料結構來管理可運行任務,按 vruntime 排序。 sched-design-CFS.rst:54-66 排程器總是選擇紅黑樹最左側的任務(即 vruntime 最小的任務)執行。 sched-design-CFS.rst:64-72

實作位於 kernel/sched/fair.c fair.c:61-79 ,核心排程邏輯在 kernel/sched/core.c。 core.c:1-9

支援的排程策略

CFS 實作三種排程策略: sched-design-CFS.rst:103-111

  • SCHED_NORMAL: 普通任務的預設策略
  • SCHED_BATCH: 批次處理任務,較少搶佔
  • SCHED_IDLE: 優先級低於 nice 19 的任務

Notes

除了 CFS 之外,Linux 還有其他排程類別用於特殊用途:RT (實時排程器) 用於 SCHED_FIFO/SCHED_RR,以及 DL (截止期限排程器)。 sched-design-CFS.rst:113 但對於一般的使用者空間程式,CFS 是預設且最常用的排程演算法。

151. 概述
16=======
17
18CFS表示“完全公平调度器”，它是为桌面新设计的进程调度器，由Ingo Molnar实现并合入Linux
192.6.23。它替代了之前原始调度器中SCHED_OTHER策略的交互式代码。
20 
21CFS 80%的设计可以总结为一句话：CFS在真实硬件上建模了一个“理想的，精确的多任务CPU”。
22
23“理想的多任务CPU”是一种（不存在的 :-)）具有100%物理算力的CPU，它能让每个任务精确地以
24相同的速度并行运行，速度均为1/nr_running。举例来说，如果有两个任务正在运行，那么每个
25任务获得50%物理算力。 --- 也就是说，真正的并行。
26
27在真实的硬件上，一次只能运行一个任务，所以我们需要介绍“虚拟运行时间”的概念。任务的虚拟
28运行时间表明，它的下一个时间片将在上文描述的理想多任务CPU上开始执行。在实践中，任务的
29虚拟运行时间由它的真实运行时间相较正在运行的任务总数归一化计算得到。
30

332. 一些实现细节
34===============
35
36在CFS中，虚拟运行时间由每个任务的p->se.vruntime（单位为纳秒）的值表达和跟踪。因此，
37精确地计时和测量一个任务应得的“预期的CPU时间”是可能的。
38
39  一些细节：在“理想的”硬件上，所有的任务在任何时刻都应该具有一样的p->se.vruntime值，
40  --- 也就是说，任务应当同时执行，没有任务会在“理想的”CPU分时中变得“不平衡”。
41
42CFS的任务选择逻辑基于p->se.vruntime的值，因此非常简单：总是试图选择p->se.vruntime值
43最小的任务运行（也就是说，至今执行时间最少的任务）。CFS总是尽可能尝试按“理想多任务硬件”
44那样将CPU时间在可运行任务中均分。
45
46CFS剩下的其它设计，一般脱离了这个简单的概念，附加的设计包括nice级别，多处理，以及各种
47用来识别已睡眠任务的算法变体。
513. 红黑树
52=========
53
54CFS的设计非常激进：它不使用运行队列的旧数据结构，而是使用按时间排序的红黑树，构建出
55任务未来执行的“时间线”。因此没有任何“数组切换”的旧包袱（之前的原始调度器和RSDL/SD都
56被它影响）。
57
58CFS同样维护了rq->cfs.min_vruntime值，它是单调递增的，跟踪运行队列中的所有任务的最小
59虚拟运行时间值。系统做的全部工作是：使用min_vruntime跟踪，然后用它的值将新激活的调度
60实体尽可能地放在红黑树的左侧。
61
62运行队列中正在运行的任务的总数由rq->cfs.load计数，它是运行队列中的任务的权值之和。
63
64CFS维护了一个按时间排序的红黑树，所有可运行任务以p->se.vruntime为键值排序。CFS从这颗
65树上选择“最左侧”的任务并运行。系统继续运行，被执行过的任务越来越被放到树的右侧 --- 缓慢，
66但很明确每个任务都有成为“最左侧任务”的机会，因此任务将确定性地获得一定量CPU时间。
67
68总结一下，CFS工作方式像这样：它运行一个任务一会儿，当任务发生调度（或者调度器时钟滴答
69tick产生），就会考虑任务的CPU使用率：任务刚刚花在物理CPU上的（少量）时间被加到
70p->se.vruntime。一旦p->se.vruntime变得足够大，其它的任务将成为按时间排序的红黑树的
71“最左侧任务”（相较最左侧的任务，还要加上一个很小的“粒度”量，使得我们不会对任务过度调度，
72导致缓存颠簸），然后新的最左侧任务将被选中，当前任务被抢占。

1005. 调度策略
101===========
102
103CFS实现了三种调度策略：
104
105  - SCHED_NORMAL：（传统被称为SCHED_OTHER）：该调度策略用于普通任务。
106
107  - SCHED_BATCH：抢占不像普通任务那样频繁，因此允许任务运行更长时间，更好地利用缓存，
108    不过要以交互性为代价。它很适合批处理工作。
109
110  - SCHED_IDLE：它比nice 19更弱，不过它不是真正的idle定时器调度器，因为要避免给机器
111    带来死锁的优先级反转问题。
112 
113SCHED_FIFO/_RR被实现在sched/rt.c中，它们由POSIX具体说明。
114
115util-linux-ng 2.13.1.1中的chrt命令可以设置以上所有策略，除了SCHED_IDLE。

為什麼搞那麼複雜，我記得BSD那邊用的算法沒那麼複雜
Fast

您提到的是 Linux CFS 排程器與 BSD 排程器的複雜度比較。

為什麼 Linux 使用較複雜的 CFS

Linux 的 CFS 使用紅黑樹和虛擬運行時間(vruntime)來追蹤任務,確實比傳統的多級反饋佇列(Multi-Level Feedback Queue)複雜。主要原因包括:

1. 更精確的公平性保證

CFS 透過 vruntime 提供數學上精確的公平性,確保每個任務獲得與其權重成比例的 CPU 時間。傳統的優先級佇列方法(如 BSD 使用的)依賴啟發式規則,在某些工作負載下可能出現不公平的情況。

2. 可擴展性需求

Linux 需要在從嵌入式設備到大型伺服器的各種環境中運行,紅黑樹的 O(log n) 操作複雜度在任務數量增加時表現更好。

3. 支援複雜的排程需求

現代 Linux 需要支援 cgroups、容器隔離、NUMA 感知等功能。 cpusets.rst:511-531 CFS 的設計更容易整合這些功能。

BSD 的簡單方法

您說得對,傳統 BSD 排程器使用更簡單的多級反饋佇列,基於優先級和時間片。這種方法:

  • 實作簡單,易於理解
  • 對於一般工作負載表現良好
  • 但在極端情況下可能出現飢餓或不公平

Linux 的權衡

Linux 選擇了複雜度換取:

  • 更好的公平性保證
  • 更好的可擴展性
  • 更容易支援現代功能(如 cgroups)

此外,Linux 現在還支援 sched_ext,允許透過 BPF 程式載入自定義排程器。 ext.c:1-8 sched-ext.rst:196-210 如果您想要更簡單的排程邏輯,可以透過 sched_ext 實作類似 BSD 的排程器而不需要修改核心。

Notes

排程器的複雜度是一個經典的工程權衡。BSD 的簡單方法在許多情況下已經足夠,但 Linux 的使用場景更廣泛,需要更複雜的機制來滿足不同需求。sched_ext 的引入提供了靈活性,讓使用者可以根據需求選擇不同複雜度的排程策略。

196Dispatch Queues
197---------------
198
199To match the impedance between the scheduler core and the BPF scheduler,
200sched_ext uses DSQs (dispatch queues) which can operate as both a FIFO and a
201priority queue. By default, there is one global FIFO (``SCX_DSQ_GLOBAL``),
202and one local DSQ per CPU (``SCX_DSQ_LOCAL``). The BPF scheduler can manage
203an arbitrary number of DSQs using ``scx_bpf_create_dsq()`` and
204``scx_bpf_destroy_dsq()``.
205
206A CPU always executes a task from its local DSQ. A task is "inserted" into a
207DSQ. A task in a non-local DSQ is "move"d into the target CPU's local DSQ.
208
209When a CPU is looking for the next task to run, if the local DSQ is not
210empty, the first task is picked. Otherwise, the CPU tries to move a task
211from the global DSQ. If that doesn't yield a runnable task either,
212``ops.dispatch()`` is invoked.
