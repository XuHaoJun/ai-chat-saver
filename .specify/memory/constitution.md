# AI Chat Saver 專案憲章

## 核心原則

### I. 最小可行產品 (MVP)
所有功能開發必須遵循 MVP 原則：優先實現核心價值，避免非必要的功能擴展。每個功能必須能夠獨立交付價值，且可以被獨立測試與驗證。在功能規格階段，必須明確區分 P1（核心）、P2（重要）、P3（增強）優先級，僅在 P1 完成並驗證後才考慮 P2 功能。

**理由**：確保專案能夠快速交付可用的產品，避免功能膨脹與開發時間延長。

### II. 可測試性 (NON-NEGOTIABLE)
所有程式碼必須具備可測試性。功能規格必須包含可獨立測試的使用者情境，每個使用者故事必須能夠獨立驗證。測試必須在實作之前撰寫（TDD 原則），且測試必須先失敗後再實作功能。整合測試必須涵蓋：新功能契約測試、契約變更、服務間通訊、共享資料結構。

**理由**：可測試性是確保品質與可維護性的基礎，缺乏測試的程式碼無法保證正確性與穩定性。

### III. 高品質標準
程式碼必須遵循清晰的架構與命名規範，具備適當的錯誤處理與日誌記錄。所有功能必須通過測試驗證，且必須符合功能規格中定義的成功標準。程式碼審查必須驗證是否符合本憲章的所有原則。

**理由**：高品質標準確保專案的長期可維護性與可擴展性，降低技術債務累積。

### IV. 簡約設計 (避免過度設計)
禁止過度設計（overdesign）。僅實作當前需求所需的功能，遵循 YAGNI（You Aren't Gonna Need It）原則。任何複雜度增加都必須有明確的業務或技術理由，且必須在實作計劃中記錄違反簡約原則的決策與原因。

**理由**：過度設計會增加開發成本與維護負擔，簡約設計確保專案能夠快速迭代並保持可理解性。

### V. 正體中文規範
所有專案文件、註解、變數命名（如適用）、使用者介面文字、錯誤訊息、日誌訊息必須使用正體中文。技術文件、API 文件、程式碼註解必須以正體中文撰寫。僅在技術限制（如第三方 API 要求）或業界慣例（如程式語言關鍵字）的情況下才可使用英文。

**理由**：確保專案文件與程式碼的可讀性與一致性，降低理解成本。

## 開發工作流程

### 功能開發流程
1. **規格階段**：建立功能規格（spec.md），包含使用者情境、需求、成功標準
2. **計劃階段**：建立實作計劃（plan.md），包含技術研究、資料模型、契約定義
3. **任務分解**：建立任務清單（tasks.md），按使用者故事優先級組織任務
4. **實作階段**：遵循 TDD 原則，先寫測試後實作功能
5. **驗證階段**：每個使用者故事完成後必須獨立驗證

### 品質檢查點
- 功能規格必須通過憲章檢查（Constitution Check）
- 實作計劃必須記錄任何違反簡約原則的決策
- 程式碼審查必須驗證測試覆蓋率與原則遵循
- 每個使用者故事完成後必須進行獨立測試驗證

## 治理規範

### 憲章優先性
本憲章優先於所有其他開發慣例與文件。任何與本憲章衝突的實務都必須以本憲章為準。

### 修訂程序
憲章修訂必須：
1. 明確記錄版本變更理由（MAJOR/MINOR/PATCH）
2. 更新所有相關模板文件以保持一致性
3. 在憲章檔案頂部記錄同步影響報告
4. 更新「最後修訂日期」

### 版本規範
版本號遵循語義化版本規範（Semantic Versioning）：
- **MAJOR**：向後不相容的原則移除或重新定義
- **MINOR**：新增原則或章節，或重大指導方針擴充
- **PATCH**：澄清、措辭修正、錯字修正、非語義性改進

### 合規審查
所有功能規格與實作計劃必須通過憲章檢查。任何違反核心原則的決策都必須在實作計劃的「複雜度追蹤」章節中明確記錄與說明。
